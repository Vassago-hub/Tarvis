# M7 上线与运营方案

## 上线策略

第一阶段不要全站直接替换原客服，建议分三步：

1. 内部试用：产品、客服、售后、技术团队使用真实官网测试。
2. 服务页灰度：先在服务支持相关页面启用。
3. 首页低比例灰度：首页右下角 5% 到 10% 流量启用。

不建议第一版直接覆盖购买页、支付页、会员登录页。

## 替换策略

不是一刀切替换原客服，而是：

```text
AI 智能服务助手
  -> 先回答、导航、排查
  -> 解决不了再转人工
  -> 高风险场景直接转人工
```

原客服入口保留在 AI 面板里，作为兜底和用户选择项。

## 灰度阶段指标

| 阶段 | 主要看什么 | 放量条件 |
| --- | --- | --- |
| 内部试用 | P0/P1 问题 | 连续 2 轮回归无 P0 |
| 服务页 5% | 找入口成功率、高风险拦截 | 3 天无严重误操作 |
| 服务页 20% | 自助解决率、转人工质量 | 自助解决率不低于 30% |
| 首页 5% | 意图识别、用户关闭率 | 负反馈不高于 5% |
| 扩大全站 | 成本下降、满意度 | 客服团队确认有效 |

## 客服团队协作

AI 不是孤立系统，要和客服团队形成闭环。

每日关注：

- 用户最常问什么。
- AI 哪些问题没解决。
- 哪些问题频繁转人工。
- 哪些回答被用户否定。
- 哪些页面路径经常失败。

客服团队每周输出：

- 新增 FAQ。
- 错误话术纠正。
- 高频投诉类型。
- 服务政策变更。
- 人工客服建议的标准回答。

AI 团队每周更新：

- FAQ 知识包。
- 页面路径规则。
- 转人工规则。
- 风险关键词。
- 失败兜底话术。

## 知识库更新机制

建议分三层维护：

1. 固定官方信息：电话、服务页、产品分类、官方 APP、客服入口。页面变动时更新。
2. 高频 FAQ：空调、电视、冰箱、洗衣机、门锁等故障排查。每周更新。
3. 动态政策：保修政策、活动、价格、库存、维修时效。第一阶段不让 AI 直接承诺，只引导官方渠道。

每条知识应包含：

```text
标题
适用产品
用户问法
标准回答
风险提示
推荐入口
更新时间
负责人
```

## 监控告警

必须监控这些事件：

| 事件 | 告警 |
| --- | --- |
| 高风险动作未确认 | P0 告警 |
| 提交动作被执行 | P0 审计 |
| 敏感字段自动填写 | P0 告警 |
| 登录/支付页面继续操作 | P0 告警 |
| 同一路径失败率突增 | P1 告警 |
| 在线客服入口打不开 | P1 告警 |
| 用户负反馈升高 | P1/P2 告警 |
| LLM 调用失败率升高 | P1 告警 |
| 响应时延异常 | P2 告警 |

## 日志与隐私

日志必须脱敏：

- 手机号：只保留后 4 位或完全脱敏。
- 地址：不记录完整地址。
- 订单号：脱敏。
- 身份证、银行卡：不记录。
- 用户输入原文可按策略采样，但涉及隐私要过滤。

建议记录：

```text
session_id
page_url
intent
action_type
risk_level
confirmation_shown
result_status
handoff_triggered
failure_reason
latency
user_feedback
```

不要记录：

```text
完整手机号
完整地址
身份证
银行卡
验证码
支付信息
账号密码
```

## 人工转接运营

转人工时，AI 应整理上下文给用户，不一定直接传给客服系统，除非后续有合规接口。

示例：

```text
问题摘要：
产品：空调
问题：显示 E1，重启后仍出现
用户已尝试：断电重启、检查滤网
建议处理：联系官方售后报修
```

用户可复制给人工客服。

## 失败复盘机制

每周看失败样本，分四类：

| 类型 | 处理 |
| --- | --- |
| 页面路径失败 | 更新 route instructions |
| 回答不准 | 更新 FAQ/知识库 |
| 用户意图没识别 | 更新 IntentRouter |
| 风险策略误杀或漏判 | 更新 SafetyPolicy |

## 组织分工

| 角色 | 职责 |
| --- | --- |
| 产品负责人 | 定义场景、优先级、体验 |
| 技术负责人 | 架构、上线、监控 |
| 客服负责人 | FAQ、转人工规则、质检 |
| 售后专家 | 故障排查内容审核 |
| 法务/隐私 | 隐私、免责声明、日志策略 |
| 品牌/市场 | 语气、视觉、对外表达 |
| 数据分析 | 指标、灰度效果、降本测算 |

## 成本与价值评估

第一阶段重点证明：

- 减少“客服入口在哪”类人工咨询。
- 减少简单故障排查人工咨询。
- 提升用户到达正确入口的速度。
- 提高转人工前的信息完整度。
- 不引入隐私和误操作风险。

## 上线节奏建议

```text
第 1 周：内部试用 + 50 条用例回归
第 2 周：服务页 5% 灰度
第 3 周：服务页 20% 灰度
第 4 周：首页 5% 灰度
第 5-6 周：根据指标扩大范围
```
