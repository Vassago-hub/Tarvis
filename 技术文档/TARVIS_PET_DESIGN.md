# Tarvis 宠物精灵设计思路与设计文档

## 1. 设计背景

Tarvis 是 TCL 官网智能服务助手的宠物精灵形象，出现在页面右下角，作为用户进入 AI 服务能力的第一触点。它不是单纯的装饰元素，而是智能体状态、品牌服务感和用户操作入口的结合体。

在本项目中，智能体需要帮助用户完成售后入口查找、人工客服分流、故障排查、站内导航和商用合作入口引导。Tarvis 的设计目标是降低用户对 AI 自动操作的陌生感，让助手在视觉上更像“随时待命的服务伙伴”，同时不干扰用户浏览网页。

## 2. 命名说明

名称：Tarvis

命名思路：

- `T` 呼应 TCL 和技术服务场景。
- `arvis` 具有轻量、灵敏、助手感，读音接近一个可被记住的角色名。
- 名称比“AI 助手”“客服机器人”更具亲和力，适合作为网页常驻入口。

Tarvis 的角色定位不是人类客服，也不冒充人工客服。它代表的是“官网智能服务助手”，负责帮用户查找路径、解释下一步、提醒风险，并在必要时转人工。

## 3. 设计目标

Tarvis 的设计目标包括四点：

- 入口明确：用户能快速发现右下角的智能助手入口。
- 状态可感知：通过动画状态表达 idle、running、review、waiting、failed 等智能体运行状态。
- 品牌亲和：视觉应贴近 TCL 服务助手的科技、可靠、温和，而不是游戏化过强。
- 低干扰：常驻页面但不遮挡核心内容，不用大面积弹窗打断用户。

## 4. 用户体验定位

Tarvis 在产品体验中的定位如下：

| 维度 | 设计取向 |
| --- | --- |
| 角色 | 官网智能服务伙伴 |
| 语气 | 可靠、主动、克制、清晰 |
| 行为 | 点击后打开对话，不主动强打扰 |
| 情绪 | 轻微拟人化，但不夸张 |
| 风险边界 | 高风险动作前提醒、确认或转人工 |

Tarvis 应该让用户感觉“这里有一个可以帮我找入口、解释流程、提醒风险的助手”，而不是“一个会擅自替我提交信息的自动化脚本”。

## 5. 视觉设计思路

### 5.1 形象原则

Tarvis 的视觉形象应满足：

- 小尺寸可读：在 40px 到 52px 的展示尺寸下仍能看清轮廓。
- 轮廓简洁：避免复杂细节、文字、徽标和过多装饰。
- 动作清楚：每个动画状态要通过姿态变化表达含义。
- 品牌关联：可以使用 TCL 服务场景中的科技感、红色点缀、柔和光泽，但不直接复制品牌 Logo。
- 透明背景：适配不同网页区域，避免白底或黑底破坏页面融合。

### 5.2 色彩方向

建议色彩系统：

| 色彩 | 用途 |
| --- | --- |
| TCL 红点缀 | 体现品牌识别和行动状态 |
| 深灰/墨色轮廓 | 保持小尺寸可读性 |
| 浅灰/银白主体 | 传达科技、清洁、服务感 |
| 柔和蓝/青辅助色 | 表达智能、扫描、思考等状态 |

色彩应克制使用，避免全身高饱和红色导致视觉压力过大。

### 5.3 风格方向

Tarvis 适合采用“轻量科技宠物”风格：

- 介于智能小助手和品牌服务精灵之间。
- 具有亲和表情，但不使用夸张卡通表演。
- 用头部、身体倾斜、手势和轻微弹跳表达状态。
- 不依赖文字气泡、感叹号、复杂道具来传达含义。

## 6. 动画状态设计

当前项目使用 8 列、9 行的 spritesheet，源帧为 192px x 208px，运行时约按 1/4 比例展示为 48px x 52px。状态行设计遵循 Codex pet 兼容结构。

| 状态 | 行为含义 | 设计建议 |
| --- | --- | --- |
| idle | 待命 | 轻微呼吸、眨眼、身体小幅浮动 |
| running-right | 横向移动/拖拽方向 | 向右运动姿态，保持节奏清晰 |
| running-left | 横向移动/拖拽方向 | 向左运动姿态，与右移动保持一致性 |
| waving | 打招呼 | 使用手/爪动作表达欢迎，不加飘散线条 |
| jumping | 活跃反馈 | 通过身体上下位移表达跳跃，不使用落地尘土 |
| failed | 出错/失败 | 轻微沮丧或停顿，避免过强负面情绪 |
| waiting | 等待用户确认 | 期待、询问、停顿姿态，强调需要用户参与 |
| running | 正在执行任务 | 表达思考、扫描、处理，不表现为真正奔跑 |
| review | 正在分析/复核 | 低头、聚焦、观察姿态，体现“正在判断” |

### 6.1 状态与智能体行为映射

| 智能体行为 | Tarvis 状态 |
| --- | --- |
| 空闲待命 | idle |
| LLM 思考或页面分析 | review |
| 工具执行中 | running |
| 等待用户确认 | waiting |
| 任务失败或被阻断 | failed |
| 用户点击入口 | idle 或 waving |

这种映射让用户能在不阅读详细文本的情况下感知助手当前状态。

## 7. 交互设计

### 7.1 常驻入口

Tarvis 默认停靠在页面右下角，作为轻量入口存在。用户点击 Tarvis 后打开助手面板，面板关闭时不销毁智能体入口，保持“可再次召唤”的体验。

### 7.2 不使用关闭宠物的交互

Tarvis 是服务入口，不应提供显眼的“关闭宠物”动作。面板可以收起，但宠物入口应保留，避免用户误关后找不到智能助手。

### 7.3 点击反馈

点击 Tarvis 后应直接打开输入区域，并聚焦输入框。这样用户的行为路径非常短：发现 Tarvis -> 点击 -> 输入问题。

### 7.4 可访问性

Tarvis 需要具备可访问名称，例如：

```text
打开 Tarvis 智能助手
Open Tarvis Assistant
```

这让屏幕阅读器用户也能理解该入口的作用。

## 8. 技术实现说明

### 8.1 资源位置

当前 spritesheet 资源保留原文件路径：

```text
packages/ui/src/assets/tcl-ray-spritesheet.webp
packages/website/public/tcl-ray-spritesheet.webp
```

虽然文件名仍包含历史名称 `tcl-ray`，但产品命名和 UI 可访问名称已统一为 Tarvis。保留资源文件名可以避免不必要的大范围路径迁移。

### 8.2 UI 接入位置

核心接入文件：

```text
packages/ui/src/panel/Panel.ts
packages/ui/src/panel/Panel.module.css
packages/website/src/components/AiAssistantButton.tsx
```

`Panel.ts` 负责宠物默认名称、点击展开面板、状态切换。`Panel.module.css` 负责 spritesheet 行列播放、尺寸、停靠位置和状态动画。`AiAssistantButton.tsx` 负责网站测试页面中的独立 Tarvis 入口。

### 8.3 尺寸与播放

运行时关键参数：

```text
源帧尺寸：192px x 208px
展示尺寸：约 48px x 52px
列数：8
行数：9
播放方式：CSS steps 动画
```

通过 `background-position` 切换行，通过 `steps(8)` 播放列帧，可以在不引入额外动画库的情况下保持轻量。

## 9. 与智能体安全策略的关系

Tarvis 的状态不是单纯视觉反馈，它应配合智能体安全策略：

- 普通导航时表现为 running 或 review。
- 需要用户确认时表现为 waiting。
- 高风险动作被阻断时表现为 failed，但文案需要解释原因。
- 转人工时保持克制，不表现为“任务失败”，而是服务流程升级。

这能帮助用户理解：智能体并不是卡住了，而是在保护用户隐私和操作安全。

## 10. 后续优化方向

| 方向 | 建议 |
| --- | --- |
| 命名统一 | 后续可将 `tcl-ray-spritesheet.webp` 逐步迁移为 `tarvis-spritesheet.webp` |
| 状态丰富 | 增加更明确的确认、转人工、成功完成状态 |
| 品牌一致性 | 进一步统一 Tarvis 与客服面板的色彩、圆角、阴影和动效节奏 |
| 视觉 QA | 使用 contact sheet 和 GIF 预览检查每行动画是否稳定 |
| 可访问性 | 为不同状态提供更清晰的 aria 状态描述 |
| 性能 | 继续使用单张 WebP spritesheet，避免多图加载和布局抖动 |

## 11. 验收标准

Tarvis 的设计和实现应满足以下标准：

- 用户能在页面上快速发现智能助手入口。
- 点击 Tarvis 后能打开助手面板并输入问题。
- 空闲、执行、等待、复核、失败状态能被动画区分。
- 宠物入口不遮挡页面主要内容。
- 小尺寸展示时轮廓清楚、没有明显裁切或错位。
- 不出现可读品牌 Logo、文字气泡或影响透明背景的白底/黑底。
- 高风险动作前的等待状态和确认文案保持一致。

## 12. 总结

Tarvis 是 TCL 官网智能服务助手的人格化入口。它的价值不是制造强烈娱乐感，而是用轻量、稳定、可感知的方式降低用户使用 AI 网页智能体的心理门槛。通过常驻入口、状态动画和安全确认流程，Tarvis 将“网页自动化能力”转化为用户更容易理解和信任的服务体验。
